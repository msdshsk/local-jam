# local-jam 設計メモ (v3)

> FigJam（部品の手作りが手間）と Figma（過剰）の中間を埋める、Webアプリの超簡単モック作成ツール。
> コンセプト = **「話しながら使える落書き帳」**（素早さ > 作り込み）。

---

## 0. 確定している前提

- スタック: **Electron + React + Tailwind + React-Flow(@xyflow/react)**
- 見た目: **ローファイ基調**（あえてラフ＝固定概念を生まない）。ただし**フォーム部品は「それが何か一目で分かる体裁」は死守**
- キャンバス: **無限キャンバスに複数画面を自由配置**（FigJam的）。画面遷移へのこだわり無し
- 画面同士の線: **非クリックの注釈線のみ**（クリックで飛ぶ遷移は作らない）
- 出口: **ローカル保存 ＋ PNG/PDF 書き出し**。共有は画面共有が主。**HTML/コード化はしない**
- 不要: フロー図特化・複数人コラボ/コメント。簡易メモの出し入れで十分

### 想定される実際の使われ方（v2で明確化）
**新機能の検討フェーズで、営業職など「要件定義に不慣れな"使う側"の人」が、画面を見ながらあーだこーだ・ガチャガチャ動かす**のが本質。
→ きれいな部品より、**「いつもの画面の骨組み」を素早く出して並べ替える**ことが価値。

**さらに重要な前提**: 対象は **ゼロからの初動構築ではなく、既に稼働しているシステムへの「機能追加」が中心**。会話の焦点は常に「追加する機能部分」。
→ 含意:
- **アプリ初動系の部品（ログイン/新規登録/オンボーディング/LP）は出番が少ない** → P2。
- **機能を構成する部品（データ入力/編集フォーム・検索＋絞り込み・一覧/テーブル・作成/編集/確認モーダル・アクションボタン）が主役** → P1。
- **テンプレート＝「既存システムのいつもの画面（ヘッダ/フッタ/サイドバー）」を背景として素早く敷き、その中に追加機能を置く**ための土台。だからマスト。

### 作る価値の核（=ここに投資する）
1. **再利用できるレイアウトテンプレート（組込み＋ユーザー定義）** ← **最重要・マスト**。適当に作った外形をテンプレ保存→取り出し。
2. **編集可能なローファイWeb部品ライブラリ**（フォーム/ヘッダ/フッタ/ナビ等）。
- キャンバス/線/保存/出力は既存ライブラリで調達し、ここで消耗しない設計にする。

## 1. データモデル

ドキュメント全体を1つの JSON として保持・保存する。React-Flow の `nodes` / `edges` をほぼそのまま採用。

```ts
// document.json = .ljam パッケージ(ZIP)内のメインファイル（§1.1参照）
interface JamDocument {
  version: number;                  // スキーマ版。マイグレーション用
  meta: { name: string; createdAt: string; updatedAt: string };
  viewport: { x: number; y: number; zoom: number };  // 開いた時の表示位置
  nodes: JamNode[];
  edges: JamEdge[];
}

// ノード（画面フレーム / 部品 / メモ を type で区別）
interface JamNode {
  id: string;
  type: NodeType;                   // 'screen' | 部品type | 'note'
  position: { x: number; y: number }; // 親があれば親からの相対座標
  width?: number; height?: number;
  parentId?: string;                // 部品 → 所属する screen の id
  extent?: 'parent';                // 部品は親フレーム内に拘束
  data: Record<string, unknown>;    // 部品ごとの編集可能な中身（後述）
}

// エッジ = 注釈線（遷移ではない）
interface JamEdge {
  id: string;
  source: string; target: string;
  type: 'annotation';               // 非クリック。dashed/手書き風
  label?: string;                   // 「保存後ここへ」等の一言
}
```

### ノードの種類

| type | 役割 | 親子 | 補足 |
|---|---|---|---|
| `screen` | **画面フレーム**（1画面=1フレーム） | 親 | サイズはプリセット（§下記）。**作成後もサイズ変更可**（ハンドル/インスペクタ数値）。`data` に画面名・デバイス種別 |

**フレーム寸法プリセット（確定）**: デスクトップ標準 **1440×1024（既定）** / デスクトップ広 1920×1080 / タブレット 768×1024 / モバイル 390×844（375×812 旧型も可） / カスタム（任意）。ローファイ前提でピクセルパーフェクトは狙わない。**いつでもサイズ変更できることが前提**。
| 部品各種 (`input`, `header` …) | **Web部品** | 子（`parentId`=screen） | フレームに `extent:'parent'` で拘束。中身は `data` |
| `note` | **付箋メモ / 一口コメント** | フリー or 部品に紐付け | トグルで一括表示/非表示。落書き帳の「メモ」機能 |

### 確定した設計判断

1. **部品はフレーム内で自由配置（絶対座標）＝ペタペタ置ける。auto-layout はしない。**
   - 「落書き帳」の素早さ優先＝Figmaのレイアウト沼を避ける。
   - **ただし整列機能は将来追加する前提で“拡張余地”を残す**：座標は常に絶対値で保持しつつ、ドラッグ確定時に座標を加工できる **「配置補助フック（placement hook）」** の差し込み点を1か所に集約しておく。これにより snap-to-grid / 整列ガイド / ペイン内整列 などを**コアの配置ロジックを作り直さずに後付け**できる。初期は素通し（フックなし）。
2. **「複合部品=1ノード」はやめる。部品は3種別（v3で確定）:**
   - **アトム**: 単一ノードの最小部品（入力/ボタン/チェック/ラベル/画像枠 等）。フリーハンドで1つずつ配置。
   - **バンドル**: ヘッダ/フッタ/フォーム/「ラベル＋入力」/CRUD骨組み 等。**置くと"個別アトムの集まり"に展開**＝最初からバラけており即・1つずつ編集できる。固めない＝**「分解」機能は不要**。実体は §2 のテンプレ/バンドル機構で扱う。
   - **データノード**: テーブル/カード 等、内部構造（行列など）をインスペクタで編集する単一ノード（原子配置に不向きなので固める）。
   - → 「1ドロップで丸ごと置ける」＋「置いた瞬間からバラバラに微調整できる」＝FigJamの手作り問題への直球回答（旧"1ノード+分解"案を上書き）。
3. **保存は `.ljam` パッケージ（ZIP／OOXML方式）**。サーバー無し。詳細は §1.1。
4. React-Flow の注意点: 子ノードは配列内で**親より後**に並べる必要がある／`expandParent` でフレーム端ドラッグ時に拡張可。

### 1.1 保存パッケージ `.ljam`（確定）
xlsx/OOXMLと同じ **ZIPパッケージ**。将来リソース（画像等）も同梱できるよう最初からコンテナ化する。
```
mydoc.ljam (zip)
├── mimetype          # "application/x-local-jam"（無圧縮・先頭エントリ）※種別判定用・任意だが推奨
├── document.json     # = JamDocument（version/meta/viewport/nodes/edges）
├── thumbnail.png     # 一覧・最近使ったファイル用プレビュー（任意）
└── resources/        # 画像等のバイナリ資産（将来用）
```
- ノードからの資産参照は **`resources/...` の相対パス**（base64でJSONを膨らませない）。
- `document.json` の `version` でスキーマ管理＋読込時マイグレーション。
- ZIP入出力は `jszip`（pure JS）想定。中身はJSONなので、いざとなれば解凍してエディタで読める（ロックインしない）。

### 1.2 オートセーブ／復帰（確定・軽量）
- 編集を **debounce** して、userData の復帰スナップショットに **プレーンJSONで上書き**（毎回ZIP化しない＝処理を軽く保つ）。
- 次回起動時、未保存の復帰データがあれば「**前回の続きを復元しますか？**」。
- クリーン保存／正常終了で復帰データをクリア。**差分・履歴は持たない**（複雑処理なし）。
- 明示保存（`.ljam`生成）は従来どおりユーザー操作。

## 2. テンプレート / バンドル機能（最重要・マスト要件）

**アトム（個別部品）を主役にしつつ、便利な並びを"束ね"て使い回す**仕組み。テンプレートとバンドルは**同じ仕組みで大きさ・用途が違うだけ**（「ラベル＋入力」のような小さな束 〜 画面まるごと）。ドキュメント(.ljam)とは別にアプリ横断で使い回せることが肝。

### 2.1 仕組み（確定）
- **作成 = ドラッグ選択 →「選択範囲をテンプレ/バンドルとして保存」**（ボトムアップ。Web技術や"いつものテンプレ"の前提が固まらない用途に最適）。
- **座標 = バウンディングボックス相対**:
  ```
  保存: 選択ノードのbbox左上(MIN)を原点(0,0)とし、各ノードを「左上からのpxオフセット」で保存
        （実寸保持＝0..1正規化はしない）。例) 絶対(520,300)(520,360)(700,300) → (0,0)(0,60)(180,0)
  取出: ドロップ地点 + 各オフセット で新IDを振り再生成。
  ```
- **展開 = アトムの集まりで降ってくる**（バンドルは固めない）。置いた瞬間から1つずつ編集可＝「分解」操作は不要。
- **フレームへの吸着**: フレーム無しの束（例：ヘッダ束）を画面フレーム上にドロップすると、**そのフレームの子として吸い込まれる**（`parentId`付与）。空白へ落とせば単独。
- screen や データノード を含む選択も同じ仕組みで保存・展開（screenを含む束＝フレームテンプレ）。

### 2.2 組込みライブラリ（ティア別・確定・最初から入れる）
すべて §2.1 と同じ仕組み。サイズで4ティア:

- **マイクロバンドル**（フォームを素早く組む最小便利単位）: ラベル＋入力 / ラベル＋セレクト / ラベル＋テキストエリア / 保存・キャンセルのボタン対 / 検索欄＋ボタン …
- **構造バンドル**: ヘッダ / フッタ / サイドバー・ナビ / パンくず / タブ / ヒーロー
- **機能バンドル**: データ入力・編集フォーム / 検索＋絞り込み / モーダル（作成・編集・確認）/ モバイル上部バー・下部タブバー・ドロワー
- **フレームテンプレ**（構造・機能バンドルを組み合わせた大きな束。下記7種）:
  1. **空フレーム**（デスクトップ / モバイル）
  2. **ヘッダ＋フッタ＋1ペイン**（最頻出の単一コンテンツ画面）
  3. **ヘッダ＋サイドバー＋コンテンツ（2ペイン）**（業務/管理画面系・最頻出）
  4. **ヘッダ＋フッタ＋2カラム**
  5. **モバイル: 上部バー＋コンテンツ＋下部タブバー**
  6. **一覧画面（CRUD）**: 2ペイン ＋〔検索＋絞り込み｜テーブル｜ページネーション〕
  7. **詳細／編集画面（CRUD）**: 2ペイン ＋〔データ入力・編集フォーム｜アクションボタン（保存/キャンセル）〕
- **アプリ初動系バンドル（P2）**: ログイン / 新規登録 / 問い合わせ / オンボーディング / LP（機能追加では出番少）

**P1で最低限揃える組込み**: マイクロ（ラベル＋入力 / ボタン対 / 検索欄＋ボタン）／構造（ヘッダ / フッタ / サイドバー）／機能（データ入力フォーム / 検索＋絞り込み / モーダル）／フレーム7種。残りは P2。

### 2.3 ユーザー定義（マイテンプレート）
- ドラッグ選択 → 名前を付けて保存（§2.1の仕組み）。取り出すと新IDのコピーがアトム展開（元は不変）。
- **管理UIのP1範囲（確定）**: 保存（名前）／一覧・取り出し／削除／リネーム ＋ サムネ（`html-to-image`で安く作れる範囲。重ければ文字ラベルのみP2）。カテゴリ/フォルダ・検索/並べ替え・インポート/エクスポートは **P2**。

### 2.4 保存の置き場所（分離）
| 種類 | 置き場所 | スコープ |
|---|---|---|
| ドキュメント | `.ljam`（ユーザー選択場所・§1.1のZIP） | その作品だけ |
| マイテンプレ / 組込み | Electron userData ストア（資産付きは`.ljam`同様パッケージ、骨組みのみ軽量JSON） | 全ドキュメントで共有 |

```ts
interface JamTemplate {
  id: string; name: string;
  tier: 'micro' | 'structure' | 'feature' | 'frame';
  category?: string; thumbnail?: string;
  nodes: JamNode[];        // position = bbox左上からのpxオフセット（実寸保持・0..1正規化しない）
  edges: JamEdge[];
  includesFrame: boolean;  // screenノードを含むか
  createdAt: string;
}
// 取出: ドロップ地点 + 各オフセットで再配置・id全採番。
//       フレーム上ドロップ かつ includesFrame=false → そのフレームの子に吸着(parentId)。
```

> 組込みもユーザー定義も**同じ `JamTemplate` 形式**（組込み=同梱初期データ、ユーザー=userData）。展開ロジックは共通化。

## 3. 部品カタログ（アトム＋データノード）

部品は **レジストリ方式**で管理。各部品 = `{ type, category, label, defaultData, インスペクタ定義, 描画コンポーネント }` の1エントリ。**拡充 = エントリ追加だけ**。

### 3.0 部品の3種別（v3で確定）
| 種別 | 例 | 振る舞い | 置き場所 |
|---|---|---|---|
| **アトム (A)** | 入力 / ボタン / チェック / ラベル / 画像枠 | 単一ノードの最小部品。フリーハンドで1つずつ配置 | 左パレット「部品」タブ |
| **データノード (D)** | テーブル / カード | 内部構造（行列など）をインスペクタ編集する単一ノード | 左パレット「部品」タブ |
| **バンドル** | ヘッダ / フッタ / フォーム / ラベル＋入力 / CRUD骨組み | 置くとアトム群に**展開**（固めない＝分解不要） | 左パレット「テンプレ/バンドル」タブ（§2） |

→ 本カタログ（§3）は **A / D のみ**を列挙。**バンドルは §2.2 の組込みライブラリ**を参照。
凡例: 種別 **A**=アトム / **D**=データノード。優先度 **P1**=MVP / **P2**=拡充。
> 優先度の軸: **「既存システムへ機能追加するとき実際に触る部品」がP1**。アプリ初動系（ログイン等）はバンドル側でP2。

### レイアウト構造のアトム
| 部品 | 種別 | 優先 | data 例 |
|---|---|---|---|
| コンテンツペイン枠 | A | **P1** | ラベル（"main"等）。2ペイン等の中身領域 |
| 2分割 / カラムコンテナ | A | **P1** | 分割数, 比率 |
| 区切り線 / セクション見出し | A | P2 | text |

### フォームのアトム
| 部品 | 種別 | 優先 | data 例 |
|---|---|---|---|
| テキスト入力 | A | **P1** | label, placeholder, required |
| テキストエリア | A | **P1** | label, rows |
| セレクト / ドロップダウン | A | **P1** | label, options[] |
| チェックボックス（群） | A | **P1** | label, items[] |
| ラジオ（群） | A | **P1** | label, items[] |
| トグル / スイッチ | A | **P1** | label, on |
| 検索ボックス | A | **P1** | placeholder |
| ボタン | A | **P1** | label, variant |
| ラベル（フォーム見出し） | A | **P1** | text |
| 日付 / ファイル / スライダー / ステッパー | A | P2 | — |

### データ表示
| 部品 | 種別 | 優先 | data 例 |
|---|---|---|---|
| 画像プレースホルダ | A | **P1** | サイズ（「×印の枠」） |
| テーブル | D | **P1** | columns[], rows数 |
| カード（単体） | D | **P1** | 画像枠, title, text, action |
| リスト / バッジ / アバター / KPI / グラフ枠 / ページネーション | A/D | P2 | — |

### 注釈 / 落書き（キャンバスレベル）
| 部品 | 種別 | 優先 | 補足 |
|---|---|---|---|
| 付箋メモ | A | **P1** | `note`。トグルで一括表示/非表示 |
| 一口コメントピン | A | **P1** | 部品に紐付く小吹き出し。クリックで開閉 |
| フリーテキスト | A | **P1** | ラベル/見出し |
| 注釈線 | A | **P1** | `edge`。dashed/手書き風。label可 |

### MVP（P1）まとめ
- **アトム/データノード（§3）**: コンテンツペイン・カラム / 入力・テキストエリア・セレクト・チェック・ラジオ・トグル・検索・ボタン・ラベル / 画像枠・テーブル・カード / 付箋・コメントピン・フリーテキスト・注釈線。
- **バンドル（§2.2）**: マイクロ（ラベル＋入力・ボタン対・検索欄＋ボタン）／構造（ヘッダ・フッタ・サイドバー）／機能（データ入力フォーム・検索＋絞り込み・モーダル）／フレームテンプレ7種。
- ※ 一周体験 = 「**2ペインのフレームテンプレを置く→アトム/バンドルで中身を組む→選択範囲を自分のバンドルとして保存→別キャンバスで再利用**」。

## 4. ローファイだが識別可能な見た目

- **モノクロ基調**（白/グレー/濃グレー）。色は「最終決定の色」と誤読されないようあえて持たせない。差し色は最小限。
- **手書き風の実現（確定）= CSS＋手書き風フォント方式**（rough.js全面採用はしない）:
  - 手書き系フォント（日本語=Klee/Yomogi等、英=Gochi Hand等）＋ラフな枠（不揃いborder / わずかな傾き / dashed / 影なし / モノクロ）。
  - **部品は通常のHTML/Reactコンポーネントのまま**＝編集・操作・レジストリ設計を崩さない。
  - **既定ON（手書き風）＋ワンクリックで「カチッと版（フラットなグレー枠）」へ切替**（検討初期=ラフ、共有直前=カチッと）。あえて強めにラフ＝固定概念を抜く狙い。
  - 将来、注釈線・区切り・画像枠の×印など"飾り図形"に限ってrough.jsを部分導入する余地は残す（コアは作り直さない）。
- **識別性の担保**: 各部品は形で何かが分かる（入力=ラベル＋プレースホルダ入り枠、セレクト=▾付き枠、チェック=小四角、画像枠=×印…）。
- **編集導線**: ダブルクリック=インライン文字編集 / 右インスペクタ=構造化項目（options[]・行数・variant 等）。

- 左パレットは **「テンプレ/バンドル（組込み / マイ）」** と **「部品（アトム/データノード）」** の2タブ構成。
- ドロップ判定: フレーム上 → そのフレームの子（`parentId`）。空白 → 単独ノード（許容）。
- **「テンプレートとして保存」**: 選択中のフレーム/ノード群を `JamTemplate` 化して userData ストアへ。

## 6. 実装アーキテクチャ

- **Electron**: main（ウィンドウ/メニュー/ファイルダイアログ/fs/**userDataのテンプレストア**）＋ preload（contextBridge で doc保存/開く・テンプレCRUD を公開）＋ renderer（Reactアプリ）。
- **Renderer**: Vite + React + TypeScript + Tailwind + `@xyflow/react`。
- **状態管理**: Zustand（React-Flow 推奨）。Undo/Redo は zundo か手動スナップショット。
- **部品レジストリ**（差別化の土台その2）:
  ```ts
  interface PartDef {
    type: string; category: string; label: string;
    defaultSize: { w: number; h: number };
    defaultData: Record<string, unknown>;
    inspector: InspectorField[];     // 右パネルの編集項目定義
    Component: React.FC<PartProps>;  // 描画（Tailwind + 手書き風）
    kind: 'atom' | 'dataNode';       // §3.0（バンドルはJamTemplate側＝§2）
  }
  ```
  → `nodeTypes` 登録もパレット生成もインスペクタもこの配列から自動構築。**部品追加 = 1エントリ追加**。
- **テンプレ/バンドルストア**（差別化の土台その1）: 組込み＋ユーザー定義を同じ `JamTemplate` 形式で保持（userData 配下、組込みは同梱初期データ）。展開＝bbox相対オフセット＋ドロップ位置で再配置・id再採番・フレームへ吸着。
- **配置補助フック**: ドラッグ確定→座標確定の間に1つの純関数フックを噛ませ、将来 snap/整列を差し込む差込口にする（初期は恒等関数）。
- **エクスポート**: PNG = `html-to-image` (toPng)、PDF = `jsPDF`。React-Flow 公式のエクスポート例に準拠。
- **保存/読込**: 明示保存=`document.json`＋資産を **`jszip`でZIP化→`.ljam`書き出し**（IPC経由でmainがfs）。読込=解凍→`document.json`パース→`version`マイグレーション。**オートセーブ=userDataにプレーンJSONをdebounce上書き**（軽量・復帰用）。

## 7. 想定マイルストーン

- **M0 雛形**: Electron+Vite+React+Tailwind+React-Flow が起動し、空キャンバスにフレーム1枚を置ける。
- **M1 最小動作**: フレームに部品（テキスト入力等）をドロップ→インライン編集→`.ljam`保存/読込。
- **M2 ★差別化体験（テンプレ一周）**: 組込み「2ペイン」テンプレを置く→中身をガチャガチャ→**「マイテンプレートとして保存」→別キャンバスで再利用**まで一周できる。
- **M3 落書き帳**: 複数フレーム＋注釈線＋付箋/コメントのトグル。
- **M4 出口**: PNG/PDF 出力。
- **M5 拡充**: P2のアトム/バンドル（アプリ初動系バンドル＝ログイン等、日付/ファイル入力 等）を順次追加。整列補助をフックに実装。
